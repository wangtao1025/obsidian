# Python 模块、包与导入机制

**适合什么时候看**：开始把代码拆成多个文件和目录，或者对 `import`、包、相对导入、`__init__.py`、`python -m` 还有点模糊时。  
**学完能做什么**：能分清模块和包，知道 Python 怎么找模块，理解常见导入写法、`__main__.py` 的作用，以及工程里应该怎么组织导入。

## 一、什么是模块，什么是包

### 1.1 模块（module）

**一句话先懂**：一个 `.py` 文件，通常就可以看作一个模块。

例如：

```text
utils.py
```

里面可以放：
- 函数
- 类
- 常量
- 顶层执行代码

### 1.2 包（package）

**一句话先懂**：一个用来组织多个模块的目录结构，通常包含 `__init__.py`。

例如：

```text
shop/
├─ __init__.py
├─ models.py
└─ services.py
```

### 1.3 为什么要有包

因为项目一大，单文件就不够了。你需要把代码按职责拆开：
- 数据模型放一组
- 工具函数放一组
- 业务逻辑放一组

这时包就能帮助你组织结构。

## 二、最常见的导入写法

### 2.1 `import 模块名`

```python
import math
print(math.sqrt(9))
```

特点：
- 命名空间清晰
- 看到 `math.sqrt` 就知道来自哪里

### 2.2 `from 模块名 import 名字`

```python
from math import sqrt
print(sqrt(9))
```

特点：
- 写起来短
- 但名字来源没那么明显

### 2.3 起别名

```python
import numpy as np
from collections import namedtuple as NT
```

别名常用于：
- 名字太长
- 社区有习惯写法

### 2.4 不建议滥用 `from xxx import *`

因为它会带来：
- 名字来源不清
- 容易冲突
- 可读性下降

## 三、`import` 到底发生了什么

### 3.1 初学者先记这个顺序

当你 `import some_module` 时，Python 通常会做这些事：
1. 看这个模块是否已经加载过
2. 没加载过就去搜索路径里找
3. 找到后执行模块顶层代码
4. 把模块对象缓存起来

### 3.2 为什么“导入模块”会执行代码

因为模块不只是“文件路径”，它也是一个运行单元。
模块顶层写的代码，在第一次导入时通常会执行。

所以工程里一般不建议在模块顶层写过重的副作用代码。

### 3.3 `sys.modules` 是什么

它可以理解成“已经加载过的模块缓存表”。

这也是为什么：
- 同一个模块重复导入，通常不会每次都重新执行顶层代码

## 四、Python 去哪里找模块

### 4.1 常见来源

模块搜索路径通常包括：
- 当前项目目录
- 标准库路径
- 虚拟环境 / site-packages
- `PYTHONPATH` 指定的额外路径

### 4.2 `sys.path` 可以看到什么

```python
import sys
print(sys.path)
```

它展示的是当前解释器用于查找模块的一组路径。

### 4.3 为什么会报 `ModuleNotFoundError`

常见原因：
- 包没安装
- 运行目录不对
- 项目结构不对
- 用错解释器
- 相对导入写错

## 五、`__init__.py` 是干什么的

### 5.1 最基础理解

它是包里的一个特殊文件，常见作用有：
- 标识包结构
- 包初始化时执行一些代码
- 控制对外暴露的名字

### 5.2 能在里面做什么

例如：
- 导出常用对象
- 定义包版本号
- 做轻量初始化

但一般不要把很重的业务逻辑塞进去。

### 5.3 `__all__` 是什么

它常用于声明：
- `from package import *` 时，默认导出哪些名字

不过在现代项目里，`import *` 本来就不推荐大量使用，所以更多是“知道有这个机制”即可。

## 六、绝对导入与相对导入

### 6.1 绝对导入

从项目 / 包根开始写完整路径：

```python
from app.services.user_service import create_user
```

优点：
- 路径更明确
- 工程里更稳定

### 6.2 相对导入

使用 `.`、`..` 表示相对层级：

```python
from .models import User
from ..utils import helper
```

### 6.3 初学者怎么选

通常建议：
- 学习阶段先优先掌握绝对导入
- 理解包结构后再补相对导入

因为相对导入一旦层级复杂，初学者容易绕晕。

## 七、`if __name__ == '__main__'` 和模块入口

### 7.1 为什么模块里常写这句

```python
def main():
  print('run')

if __name__ == '__main__':
  main()
```

因为一个模块既可能：
- 被直接运行
- 也可能被别的文件导入

### 7.2 直接运行和被导入的差别

- 直接运行：`__name__ == '__main__'`
- 被导入：`__name__` 是模块名

所以这句判断可以控制：
- 模块被导入时不执行演示代码
- 模块直接运行时才进入入口逻辑

## 八、`python -m` 和 `__main__.py`

### 8.1 `python -m` 是按模块方式执行

```bash
python -m http.server
```

这表示：
- 让 Python 按模块机制找到并运行这个模块

### 8.2 包里的 `__main__.py`

如果一个包里有：

```text
mypkg/
├─ __init__.py
└─ __main__.py
```

那么可以这样运行：

```bash
python -m mypkg
```

此时会执行包中的 `__main__.py`。

### 8.3 这对工程有什么意义

这让“一个包本身就是可运行入口”成为可能，CLI 工具、脚本包都很常见。

## 九、工程里怎么组织模块和包更清晰

### 9.1 小项目

```text
project/
├─ app.py
├─ utils.py
└─ config.py
```

### 9.2 稍大一些的项目

```text
project/
├─ app/
│  ├─ __init__.py
│  ├─ main.py
│  ├─ models/
│  ├─ services/
│  └─ utils/
└─ tests/
```

### 9.3 最重要的组织原则

- 一个模块只做一类事情
- 不要让文件越来越“万能”
- 导入方向尽量清晰，避免循环依赖

## 十、最常见的导入问题

### 10.1 循环导入

A 导入 B，B 又导入 A，就容易出问题。

常见解决思路：
- 重新拆模块职责
- 把公共部分抽出去
- 延迟导入

### 10.2 直接运行包内文件导致相对导入报错

这是初学者很常见的问题。

原因通常是：
- 你把一个包内部文件当脚本直接跑了
- 但它原本是按包结构设计的

这时往往更适合用：

```bash
python -m 包名.模块名
```

### 10.3 名字冲突

例如你自己写了个 `random.py`，就可能和标准库 `random` 冲突。

所以文件名不要乱取成和标准库、第三方库重名。

## 十一、本章高频问法速记

- **模块是什么**：一个 `.py` 文件通常就是一个模块。
- **包是什么**：组织多个模块的目录结构，通常有 `__init__.py`。
- **`import` 做了什么**：查缓存、找模块、执行顶层代码、缓存模块对象。
- **`__init__.py` 有什么用**：定义包结构、轻量初始化、控制导出。
- **`python -m` 有什么意义**：按模块 / 包机制运行，更适合工程入口。
- **绝对导入和相对导入怎么选**：工程里一般更推荐优先使用绝对导入。

---

**本章小结**：模块和包不是“语法点小知识”，而是 Python 工程结构的核心。学会 `import`，本质上是在学“代码如何拆分”“解释器如何查找模块”“项目如何形成清晰边界”。把这些关系理顺之后，后面学框架、写服务、做自动化项目都会顺很多。
